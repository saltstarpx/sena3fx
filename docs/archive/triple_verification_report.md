### 結果

今回の再現（v77・15M・半決済あり）は、採用報告の数値と**概ね整合**するが、一部の差異がある。

| 指標 | 採用報告 | 今回再現 | 差分 |
|------|--------:|--------:|-----:|

### 差異の原因分析

**トレード数の差（-20本）:** 採用報告は `qlib_factor_backtest.py` でv76の全シグナルにKMID+KLOWフィルターをかけた結果（327本）。今回は `yagami_mtf_v77.py` のgenerate_signals関数を直接呼び出した結果（307本）。後者は1H足を15分足から集約する際の境界条件が若干異なる。

**勝率・PFの差（-5.4pt / -1.50）:** 採用報告のシミュレーションはv76の15分足シグナルをそのまま使用。今回の再現はv77のgenerate_signals（1H足を15Mから集約）を使用しており、エントリータイミングが若干ずれる場合がある。

**結論:** 採用報告の数値は**再現可能な範囲内**にある。v77がv76を大幅に上回ることは今回の再現でも確認された（PF 3.46 vs 2.17）。

## 検証②：半決済あり vs 全決済の影響分析

### 半決済ロジックの仕組み

```
半決済あり（採用報告方式）:
  ① エントリー後、RR1.0のポイント（raw_ep + risk）に到達したら
     → ポジションの半分を決済（半分の利益を確定）
     → SLをブレークイーブン（raw_ep）に移動
  ② 残り半分はTP（RR2.5）まで保有 or SL（BE）で決済

全決済のみ（シンプル方式）:
  → SL or TP のどちらかに到達したら全決済
  → 中間での部分決済なし
```

### 定量的影響

| 指標 | 半決済あり | 全決済のみ | 差分 |
|------|----------:|----------:|-----:|
| トレード数 | 307 | 228 | **+79本** |
| 勝率 | 70.7% | 41.7% | **+29.0pt** |
| PF | 3.46 | 1.42 | **+2.04** |
| 総損益 | +10,540 | +2,502 | **+8,038** |
| MDD | 398.7 | 805.3 | **-406.6** |
| プラス月 | 12/12 | 7/12 | **+5ヶ月** |

### 解説

半決済ロジックが勝率・PFに与える影響は**極めて大きい**。

**トレード数が増える理由:** 半決済後にSLがBEに移動するため、「半分TP→残り半分BE決済」というパターンが発生し、1エントリーが2回の決済記録になる場合がある（実装依存）。

**勝率が+29ptになる理由:** RR1.0のポイントで半決済することで、「小さな勝ち」が多数発生する。全決済では「SL or TP」の二択なので、勝率はRR2.5の理論値（約28-30%）に近くなる。半決済では「RR1.0で勝ち」が追加されるため見かけ上の勝率が大幅に上昇する。

**PFが+2.04になる理由:** BE移動によりSL時の損失が大幅に圧縮される。全決済では負けトレードの損失がフルに発生するが、半決済では「半分利確後にBE」となるため損失がほぼゼロになる。これがPFを劇的に改善する。

> **重要な注意点:** 半決済ロジックは実運用では「スリッページ」「流動性」の問題が発生しやすく、バックテストの数値がそのまま実現しない可能性がある。全決済（PF 1.42）が「保守的な下限値」、半決済（PF 3.46〜4.96）が「理想的な上限値」と捉えるべきである。

#### v77のシグナル生成（KMID+KLOWフィルター追加）

```python
# v77: KMID+KLOWフィルター関数
def check_kmid_klow(prev_4h_bar, direction):
    o, h, l, c = prev_4h_bar["open"], prev_4h_bar["high"], prev_4h_bar["low"], prev_4h_bar["close"]
    
    # KMID: 実体方向一致（ロングなら陽線、ショートなら陰線）
    kmid_ok = (direction == 1 and c > o) or (direction == -1 and c < o)
    
    # KLOW: 下ヒゲ比率 < 0.15%（モメンタム純度）
    body_bottom = min(o, c)
    klow_ratio = (body_bottom - l) / o if o > 0 else 0
    klow_ok = klow_ratio < 0.0015
    
    return kmid_ok and klow_ok

# v77: 4H足の二番底（フィルター追加）
if trend == 1:
    if abs(low1 - low2) <= tolerance and h4_prev1["close"] > h4_prev1["open"]:
        if not check_kmid_klow(h4_prev1, direction=1):  # ← 追加3行
            continue
        # 以降はv76と同一

# v77: 1H足の二番底（フィルターは直前4H足で判定）
if trend == 1:
    if abs(low1 - low2) <= tolerance and h1_prev1["close"] > h1_prev1["open"]:
        if not check_kmid_klow(h4_latest, direction=1):  # ← 1H足パターンでも4H足で判定

```

## 定量分析サマリー

### 採用報告との差異の本質

採用報告（PF 4.96）と今回再現（PF 3.46）の差は、**シミュレーション実装の細部の差異**によるものであり、v77の優位性そのものは変わらない。

1. **シグナル生成の差:** qlib_factor_backtest.pyはv76の全シグナルにフィルターをかける方式、v77.pyは生成時点でフィルターをかける方式。前者の方がシグナルが若干多い（327 vs 307）。

2. **半決済の影響:** 半決済ロジックがPFを+2.04押し上げている。これは実運用では過大評価の可能性があるため、全決済（PF 1.42）を保守的な下限として認識すべきである。

### v77の実力評価

| シナリオ | PF | 評価 |
|---------|----:|------|
| 採用報告（半決済・理想条件） | 4.96 | 上限値（過大評価リスクあり） |
| 今回再現（半決済） | 3.46 | 現実的な推定値 |
| 全決済のみ（保守的） | 1.42 | 下限値 |

**実運用での期待PFは 1.5〜2.5 の範囲**と見るのが現実的である。
